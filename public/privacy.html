Privacy Policy

Last updated: March 1, 2026

1. Our Philosophy

BitMask is built on a single principle: we don't want your data. We designed our system so that we technically cannot access, read, or store your messages — even if compelled to do so. Privacy is not a feature we bolted on; it is the architecture itself.

2. Information We Do NOT Collect

  • Messages & Files: All messages and files are end-to-end encrypted (ECDH + AES-256-GCM). The server relays ciphertext it cannot decrypt. Messages and files are never written to disk or any persistent database.
  • Personal Information: We do not collect names, emails, phone numbers, or any personally identifiable information. There is no account creation or login.
  • Chat History: Messages exist only in volatile memory during transit and self-destruct according to the timer you set (5 seconds to 24 hours). Once destroyed, they are irrecoverable.
  • Passphrases: If you use the passphrase lock feature, your passphrase is never stored. It is used to derive an encryption key via PBKDF2 (310,000 iterations), which encrypts your identity locally. Only the encrypted blob is saved in your browser.

3. Information Stored in Your Browser

BitMask offers optional identity persistence features that store data only in your own browser — never on our servers:

Data Storage When
Peer ID + encryption keys sessionStorage (default) Always — erased when you close the tab
Peer ID + encryption keys localStorage Only if you enable "Remember Identity" — persists across sessions until you clear it
Encrypted identity blob localStorage Only if you set a passphrase lock — contains your identity encrypted with your passphrase
Sound preference localStorage If you toggle sounds on/off — stores a single boolean

You can clear all browser-stored data at any time by clicking "New ID" or clearing your browser's site data. We have no access to anything stored in your browser.

4. Information Temporarily Held in Server Memory

While you are actively connected, the server holds the following in volatile RAM only (never persisted to disk):

Data Purpose Lifetime
Peer ID → Socket mapping Route messages between peers Until you disconnect or close the tab
Your public encryption key Shared with peers for key exchange Until you disconnect
Room ID + Room Code Link peers in a conversation (1:1 or group room up to 20 members) Until all peers disconnect or leave
Room membership list Track which peers are in a group room for message routing Until the peer leaves or disconnects
IP address (hashed, in memory) Rate limiting to prevent abuse Rolling window, then discarded

If the server restarts, all of this data is destroyed instantly. There is no backup, no replication, and no recovery mechanism — by design.

5. End-to-End Encryption

BitMask uses the Web Crypto API built into your browser:

  • Key Exchange (1:1): ECDH (Elliptic Curve Diffie-Hellman) on the P-256 curve — both peers derive the same shared secret independently
  • Key Exchange (Groups): The room host generates a random AES-256 group key. When a new peer joins, the host encrypts the group key using an ECDH-derived pairwise key and sends it to the joiner. The server never sees the group key.
  • Message Encryption: AES-256-GCM with a unique 96-bit initialization vector per message, using the pairwise key (1:1) or group key (rooms)
  • File Encryption: Same AES-256-GCM encryption applied to file data — images, documents, and all file types are encrypted before leaving your device
  • Identity Export: When you export your identity, it is encrypted with a passphrase-derived key (PBKDF2 + AES-256-GCM) before being saved to a file

Encryption keys are generated in your browser and never leave your device. The server only sees opaque ciphertext blobs. We have zero knowledge of your message or file content.

6. Self-Destructing Messages & Files

Every message and file has a time-to-live (TTL) set by the sender:

  • Client-side: The recipient's browser starts a countdown timer when the content is displayed. At zero, the DOM element is destroyed, blob URLs are revoked, and the plaintext is garbage-collected.
  • Server-side: Any encrypted content queued for an offline peer is purged from memory when its TTL expires. Read-acknowledgment triggers immediate deletion of any queued copy.

7. Identity Management

BitMask provides several identity features — all operating entirely within your browser:

  • Remember Identity: Optionally save your Peer ID and keys to localStorage so they survive tab close. Disable at any time to revert to session-only storage.
  • Passphrase Lock: Encrypt your stored identity with a passphrase. The passphrase is never saved — only the encrypted result is stored. Wrong passphrase = no access.
  • Export/Import: Download your identity as an encrypted .bitmask file. Import it on any device. The export file can be optionally passphrase-protected.
  • QR Code: Generate a QR code of your Peer ID for easy sharing. The QR is generated locally — no data is sent to any server.

None of these features send any identity data to our server beyond the Peer ID and public key needed for routing.

8. Group Rooms

BitMask supports encrypted group rooms for 3–20 peers:

  • Room Codes: When a room is created, a random 6-character alphanumeric code is generated. This code is used by others to join. Room codes exist only in server RAM and are deleted when the room empties.
  • Group Encryption: The room host generates a random AES-256 group key in-browser. This key is distributed to each joining member encrypted via an ECDH pairwise channel. The server never sees the group key — it only relays the encrypted blobs.
  • Membership: The server tracks which Peer IDs are in a room (in volatile RAM) to route messages. This membership data vanishes when the room empties or the server restarts.
  • Host Transfer: If the room host disconnects, host privileges automatically transfer to the next member. This is purely a routing role — the host has no special access to message content.

All group messages and files are encrypted with the shared group key before leaving your device. The server is a relay and cannot decrypt any group content.

9. Cookies & Tracking

BitMask uses zero cookies, zero analytics trackers, zero advertising pixels, and zero third-party scripts. We do not use Google Analytics, Facebook Pixel, or any similar service.

Browser storage (sessionStorage and localStorage) is used only for your identity data and preferences — never for tracking.

10. Third-Party Services

The only external resources loaded are Google Fonts (Inter and JetBrains Mono) for typography. Google's font service may log standard web request data (IP address, user agent) under their own privacy policy. No other third-party services are used. The QR code generator is fully self-contained with no external dependencies.

11. Data Requests & Law Enforcement

Because we do not store messages, files, user identities, or chat metadata on disk, we have nothing to provide in response to data requests, subpoenas, or legal orders. You cannot hand over what you don't have.

12. Children's Privacy

BitMask is not directed at anyone under the age of 13. We do not knowingly collect data from children. Since we collect no personal information from any user, this risk is inherently mitigated by our architecture.

13. Changes to This Policy

We may update this Privacy Policy from time to time. Changes will be reflected by the "Last updated" date at the top of this page. Since we have no way to contact you (we don't have your email), we encourage you to review this page periodically.
